01Intake to runtime spec
⚙ Technical

Convert the intake workflow into a runtime contract: planner objective, allowed reads/writes, blocked actions, tool boundaries, escalation owners, state tables, retry policy, audit fields, and rollback command for every side effect.

☆ What this means for you

We take everything you told us about how emergency calls, scheduling, and follow-ups work at Ally Plumbing and turn it into a precise set of rules. The "runtime spec" is the instruction manual for the agent: what it can do, what it can't do, who gets notified when something goes wrong, and how to undo any action it takes.

✓ Deliverables
Workflow scope documentAllowed/blocked action listEscalation owner mapRollback plan
⚠ Build it yourself

Map every workflow path end-to-end: what triggers it, who owns each step, what decisions get made, and what happens when something goes wrong. Write down every edge case (what if the customer calls back before the first callback completes? what if the address is outside your service area?). Then translate each path into a rule set with explicit stop conditions, escalation points, and a rollback command for every action. Expect 3-5x more edge cases than you initially estimate.

02Event taxonomy
⚙ Technical

Enumerate every trigger: missed call, voicemail, form submit, inbox reply, scheduled scan, overdue follow-up, retry, timeout, manual override. Assign source tool, schema version, payload hash, owner, priority, and correlation ID.

☆ What this means for you

We map out every type of event that should wake the agent up. A missed call from CallRail, a voicemail transcription, an emergency form, a customer reply, an overdue follow-up. Each gets tagged with where it came from and how urgent it is. This is how the agent knows a water-damage emergency at 2 AM from a routine scheduling request.

✓ Deliverables
Complete trigger listPriority classification matrixEvent schema per source
⚠ Build it yourself

Go into each platform's settings and find every webhook, notification, and form endpoint that fires. List each with its payload format. Design a unified event schema that normalizes all of them with fields for source, priority, owner, and a unique correlation ID. Build deduplication logic so the same voicemail arriving twice doesn't create two dispatch tasks. Start with CallRail's webhook docs and work outward.

03MCP capability registry
⚙ Technical

Expose only approved capabilities: lookup customer, classify urgency, draft reply, create job note, request approval, log trace, rollback. Each has auth scope, risk class, JSON schema, rate limit, and test fixtures.

☆ What this means for you

Instead of giving the agent the keys to your entire ServiceTitan account, we create a menu of specific, approved actions. "Look up a customer record" is one capability. "Create a dispatch note" is another. The agent never gets raw access to your tools. It can only use capabilities we've explicitly listed, tested, and approved.

✓ Deliverables
Capability registryPermission matrix per toolRate limit configTest fixtures
⚠ Build it yourself

For each action the agent should take (e.g., "look up customer by phone number," "create dispatch note," "check calendar availability"), write a JSON schema defining the exact input fields, output format, and error responses. Map each action to the narrowest possible OAuth scope. Set rate limits per action and write test fixtures -- sample inputs with known correct outputs -- to verify each capability works before connecting it to the agent.

04Tool adapter contracts
⚙ Technical

Wrap vendor APIs behind typed adapters handling pagination, rate limits, vendor errors, response normalization, idempotency keys, request tracing, stale-read detection, and rollback-safe writes.

☆ What this means for you

Every tool you use has its own API with its own quirks, limits, and error messages. We build a wrapper around each one that handles all that complexity. If ServiceTitan is slow or returns an error, the adapter retries safely. If the agent tries to create a duplicate record, the adapter catches it.

✓ Deliverables
Typed adapter per toolError handling rulesRetry + idempotency config
⚠ Build it yourself

Open the API documentation for CallRail, ServiceTitan, and Google Calendar. For each, build a wrapper that handles paginated responses, rate-limit headers (back off instead of crashing), error codes (map each vendor's format to a standard structure), and idempotency (retrying a timed-out write shouldn't create a duplicate). Test each wrapper independently against the live API with a sandbox account.

05Serverless ingress
⚙ Technical

Create isolated backend handlers for signed webhooks, auth callbacks, scheduled scans, queue workers, CRM/payment writes, approval callbacks, and admin retries. Every handler validates tenant, signature, schema version, and actor.

☆ What this means for you

This is the front door for incoming events. When CallRail sends a "missed call" notification, it hits a secure endpoint that verifies the message is real (not spoofed), checks the format, and queues it for the agent. Each handler is isolated, so a problem with one doesn't affect the others.

✓ Deliverables
Webhook handlers per toolSignature validationIngress monitoring
⚠ Build it yourself

Set up a serverless function (AWS Lambda, Cloudflare Workers, or similar) for each event type. Each function needs to: verify the webhook signature (CallRail uses HMAC-SHA256, ServiceTitan uses a different method), validate the payload against your schema, reject malformed requests, and enqueue valid events. Deploy each on its own endpoint so a failure in one doesn't take down the others.

06Queue and retry fabric
⚙ Technical

Transform inbound events into command envelopes: tenant, actor, target tool, correlation ID, idempotency key, retry count, backoff policy, poison-event flag, DLQ route, and replay metadata.

☆ What this means for you

Not everything can be processed instantly. The queue organizes incoming work into a reliable pipeline. If ServiceTitan is unreachable, the task waits and retries automatically. If a task fails repeatedly, it goes to a dead-letter queue for investigation without blocking everything else. Every action gets a unique ID so nothing runs twice.

✓ Deliverables
Message queue configRetry + backoff policyDead-letter queueReplay tooling
⚠ Build it yourself

Choose a message broker (AWS SQS, RabbitMQ, or Google Pub/Sub). Generate unique idempotency keys so duplicates get caught. Configure exponential backoff (1s, 4s, 16s) with a max retry count. Route repeated failures to a dead-letter queue for inspection. Build a replay tool to re-process failed events after fixing the root cause.

07State model
⚙ Technical

Persist source event, run phase, planner decision, tool trace, approval status, rollback pointer, generated draft, delivery status, customer-visible outcome, exception reason, and final resolution as separate records.

☆ What this means for you

Every time the agent handles a call or request, a complete record is saved: what came in, what the agent decided, what tools it used, what it sent, and the outcome. If a customer ever asks "what happened with my request?", you can pull the exact sequence of events in seconds.

✓ Deliverables
State schemaAudit trail per actionSearchable run history
⚠ Build it yourself

Design a database with separate tables for: source events, agent decisions, tool requests/responses, approval records, and outcomes. Link each record via a correlation ID. Build a query layer so you can ask "show me every action for this customer" or "which events failed last Tuesday." Without this, you're left with unstructured logs that can't be searched or audited.

08Retrieval pipeline
⚙ Technical

Index SOPs, service policies, price rules, service-area pages, FAQs, examples, and prior resolutions with source IDs, trust levels, freshness windows, citation rules, and stale-context guards.

☆ What this means for you

The agent needs to know your business rules: service area boundaries, pricing tiers, emergency classification, warranty policies. We index all of this with freshness dates, so the agent never references outdated policies. Think of it as a searchable, always-current employee handbook.

✓ Deliverables
Indexed policy libraryFreshness rulesCitation config
⚠ Build it yourself

Gather all reference documents: service area maps, pricing sheets, warranty policies, FAQs, past resolutions. Break each into small chunks and load into a vector database (Pinecone, Weaviate, or pgvector). Tag each with a source ID, trust level, and expiration date. Set up re-indexing whenever a document changes -- if pricing updates and the index doesn't, the agent quotes the old price.

09Permission lattice
⚙ Technical

Keep secrets server-only in encrypted token storage. Bind OAuth scopes to capability IDs. Enforce separate permissions for read, draft, write, money movement, PII, irreversible mutation, admin override, and rollback.

☆ What this means for you

Your API keys and credentials never touch the AI model. They're stored in encrypted vaults used only by server-side code. The agent has different permission levels: read a customer record freely, but creating a dispatch task requires higher permission, and anything involving payments or personal data requires explicit approval.

✓ Deliverables
Permission matrixEncrypted credential vaultApproval routing rules
⚠ Build it yourself

Set up an encrypted secret vault (AWS Secrets Manager, HashiCorp Vault, or similar). Never store credentials where the AI can access them. Configure the narrowest OAuth scopes per tool with automatic token refresh. Build a permission matrix: reads auto-approved, drafts logged, writes need elevation, money/PII needs human sign-off. Log every credential access. One misconfiguration and the agent touches data it shouldn't.

10Policy and approval engine
⚙ Technical

Risk-score planned commands using capability risk, payload fields, confidence, customer impact, dollar exposure, PII exposure, historical error rate, and approval policy before execution.

☆ What this means for you

Before the agent takes any action, a policy engine scores how risky it is. Looking up a customer record? Low risk, auto-approved. Sending a price quote? Medium, might need review. Issuing a refund? High, requires your team's approval. You set the thresholds. The agent follows them every time, with no exceptions.

✓ Deliverables
Risk scoring matrixApproval routing configAuto-allow/block rules
⚠ Build it yourself

Build a rules engine that evaluates every planned action. Define risk dimensions: capability type, dollar amount, PII presence, confidence score, error history. Assign a composite score and route: low auto-executes, medium logs for review, high pauses and notifies a human with context. Store every scoring decision in an audit table.

11Simulation harness
⚙ Technical

Generate hundreds of simulated calls, tickets, malformed payloads, duplicate events, no-availability cases, approval delays, and vendor failures. Replay golden traces and target unhandled errors toward <0.01%.

☆ What this means for you

Before the agent handles a single real customer, we run it through hundreds of test scenarios: normal calls, 3 AM emergencies, incomplete information, duplicate voicemails, ServiceTitan outages, scheduling conflicts. It only graduates to real customers after passing every test.

✓ Deliverables
Test scenario libraryGolden trace recordingsPass/fail reportRegression suite
⚠ Build it yourself

Write 200+ test scenarios: normal calls, 3 AM emergencies, incomplete info, duplicate voicemails, tool outages, calendar conflicts, out-of-area requests. Build a harness that replays each and records results. Compare against what a human should have done. Fix every failure and re-run until errors are below 0.01%. Save passing scenarios as "golden traces" to re-run after future changes.

12Rollout and rollback
⚙ Technical

Start in shadow mode, compare against staff decisions, then enable staged writes by capability. Monitor latency, approval rate, failed calls, drift, duplicate prevention, quality, SLO alerts, kill switch, and rollback.

☆ What this means for you

We don't flip a switch on day one. The agent starts in "shadow mode," watching real calls and showing what it would have done without actually doing anything. Once confirmed, we turn on capabilities one at a time. At any point, a kill switch pauses the agent instantly, and every action can be rolled back.

✓ Deliverables
Shadow-mode dashboardStaged rollout planKill switchRollback commandsSLO monitoring
⚠ Build it yourself

Deploy in shadow mode first: the agent logs what it would do without acting. Compare against staff decisions for 1-2 weeks. Turn on the lowest-risk capability via feature flag. Monitor errors, latency, and approval rates. Add capabilities one at a time. Build a kill switch that disables everything in under 60 seconds, and write rollback procedures for every write action. Test the kill switch before you need it.
